# Лицензирование ПО: как начать, как защитить

Ниже — краткий практический гид, с чего начать обсуждение лицензирования программного обеспечения, как оформить права и какие меры защиты обычно используют. Этот материал не является юридической консультацией, но поможет подготовиться к разговору с юристом.

## 1. Что нужно определить в начале

1. **Кто владеет правами**
   - Общество с ограниченной ответственностью "Корпорация Информационные Технологии".
   - Авторское право получено.

2. **Модель распространения**
   - Только **Коммерческая** (платные лицензии).

3. **Цели лицензирования**
   - Защита интеллектуальной собственности.
   - Монетизация.
   - Контроль условий использования.
   - Снижение рисков (гарантии/ответственность).

## 2. Виды лицензий (общая ориентация)

1. **Проприетарные (закрытые)**
   - Четко ограничивают способы использования.
   - Возможны варианты: per‑seat, per‑device, per‑server, per‑usage, подписка.


## 3. Как «защитить» ПО на практике

### 3.1. Юридическая защита

- **Лицензионный договор**:  
  описывает допустимые способы использования, ограничения, ответственность, порядок обновлений/поддержки.  
- **Договоры с разработчиками/подрядчиками**:  
  фиксируют переход или лицензирование прав к владельцу продукта.  
- **Политика авторских прав**:  
  укажите, кто владеет правами, и контакт для лицензирования.

### 3.2. Техническая защита

- **Лицензионные ключи** и/или **активация**.
- **Привязка лицензии** к устройству/учётной записи.
- **Онлайн‑проверки** валидности лицензии.
- **Сегментация функционала** по уровням лицензий.
- **Обфускация** (для некоторых типов приложений) — снижает риск реверса.

#### Примерная схема активации (ключ + подпись)

Ниже — упрощенная схема, как обычно строят проверку лицензии, чтобы избежать подделок:

1. **Генерация лицензии на сервере**  
   - Сервер формирует payload (ID клиента, продукт, срок, уровень).  
   - Payload подписывается приватным ключом.  

2. **Активация на клиенте**  
   - Клиент получает payload + подпись.  
   - Проверяет подпись публичным ключом.  
   - Сохраняет лицензию локально (например, в защищенном хранилище).  

3. **Проверка при запуске**  
   - Проверка подписи и срока действия.  
   - Проверка привязки к устройству/аккаунту (hash от оборудования или ID пользователя).  

Пример псевдокода проверки подписи (C#-стиль):

```csharp
bool VerifyLicense(byte[] payload, byte[] signature, byte[] publicKey)
{
    using var rsa = RSA.Create();
    rsa.ImportSubjectPublicKeyInfo(publicKey, out _);
    return rsa.VerifyData(payload, signature, HashAlgorithmName.SHA256, RSASignaturePadding.Pkcs1);
}
```

> Важно: приватный ключ хранится только на сервере. Клиентское приложение хранит лишь публичный ключ.

### 3.3. Организационные меры

- **Процессы выдачи и учёта лицензий**.
- **Система поддержки и обновлений** (доп. ценность для легальных пользователей).
- **Мониторинг злоупотреблений** и корректные процедуры реакции.

## 4. Что обычно входит в лицензионный документ

- Определения (что такое «ПО», «пользователь», «устройство» и т. п.).
- Права и ограничения (установка, копирование, распространение, модификации).
- Ограничение ответственности, отказ от гарантий.
- Условия оплаты, срок, прекращение.
- Поддержка/обновления.
- Юрисдикция и порядок разрешения споров.

## 5. С чего начать в вашем случае

1. **Опишите продукт и его аудиторию** (B2B/B2C, on‑premise/SaaS, критичные риски).
2. **Определите модель монетизации** и тип лицензии.
3. **Выберите базовую лицензию** (если OSS) или черновик EULA (если проприетарная).
4. **Проверьте цепочку прав** (кто автор, есть ли NDA и договоры).
5. **Согласуйте с юристом** финальную редакцию.
